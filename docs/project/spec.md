# Share HTML Spec

## Purpose

Share HTML lets a user upload one self-contained HTML file and get a shareable online URL. The uploaded HTML is treated as untrusted content: the product does not sanitize it into the app DOM, it stores the file privately, checks metadata before rendering, and streams the preview through an isolated route.

## Product Model

- Anonymous users can create temporary shares.
- Signed-in users can keep, view, claim, and delete their shares.
- Each upload produces a public share page and a direct preview page.
- Public shares are unlisted and addressed by random slugs.
- HTML previews may run scripts, but only inside a sandboxed iframe or preview route.
- Auth emails are generated by Supabase Auth but delivered through the Share HTML Worker and Cloudflare Email Service.

## URL Model

- `Share URL`: `/s/:slug`
  - The user-facing wrapper page.
  - Shows title, status, risk context, report action, and embedded preview.
  - This is the default URL to send to another person.
- `Preview URL`: `/v/:slug/`
  - Streams the uploaded HTML itself.
  - Used when someone wants to inspect the page without the wrapper UI.
- `Claim token`
  - Private token returned for anonymous uploads.
  - Used after sign-in to attach an anonymous share to an account.

## Architecture

```text
Browser
  |
  | app shell, /s/:slug
  v
Cloudflare Worker Static Assets
  |
  | /api/*
  v
Worker API
  |-- Supabase Auth
  |-- Supabase Postgres metadata
  |-- Cloudflare R2 private HTML objects
  |-- Cloudflare Email Service auth delivery
  |
  | /v/:slug/
  v
Sandboxed preview response
```

## Auth And Users

Supabase Auth provides magic-link sign-in and token/session handling. The app uses Supabase's Send Email Auth Hook so auth emails are sent by the Worker through Cloudflare Email Service from `Share HTML <sharehtml@zhenjia.dev>`.

Supabase Postgres stores profiles, shares, assets, events, reports, and moderation state.

The Worker is the trusted server-side actor. Browser clients receive only the publishable Supabase key; privileged metadata writes use Worker-held secrets and RLS policies.

## Upload Flow

```text
POST /api/shares
  -> optional Supabase JWT verification
  -> size/type checks
  -> SHA-256 hash
  -> lightweight risk scan
  -> create share metadata
  -> put HTML body into R2
  -> activate, review, or block the share
  -> return share URL, preview URL, and optional claim token
```

Anonymous uploads expire by default. Signed-in uploads belong to the authenticated user.

## Preview Flow

```text
GET /v/:slug/
  -> lookup share metadata
  -> reject deleted, expired, blocked, or missing shares
  -> stream private R2 object
  -> attach restrictive response headers
```

The share page embeds previews with an iframe sandbox. The preview is intentionally isolated from the main app UI.

## Moderation And Abuse Controls

The MVP uses explainable rule-based scanning:

- password forms pointing at external origins
- wallet, seed phrase, and private-key language
- external `base href`
- top-level navigation attempts
- mixed-content HTTP references
- suspicious short-link or script patterns

Risk scores decide whether a share becomes active, needs review, or is blocked.

## Deployment Model

GitHub is the release gate and Cloudflare is the deploy runner:

```text
PR -> GitHub CI Build check -> merge to protected main -> Cloudflare Workers Builds deploys main
```

Cloudflare Workers Builds is connected to `lifeodyssey/share-html` with:

- Production branch: `main`
- Build command: `npm run build`
- Deploy command: `npx wrangler deploy`
- Root directory: `/`
- Non-production branch builds: disabled

GitHub `main` is protected with a required `Build` check and PR-only updates.
